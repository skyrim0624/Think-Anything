# VidMark 产品规格

## 产品定位

VidMark 是 Think Anytime 的视频精读子系统。它以浏览器插件作为入口，把用户在 YouTube 或 X 上刷到的必看视频，转换成可精读、可跳转、可翻译、可笔记、可沉淀到 Obsidian 的中文学习材料。

VidMark 不做“值不值得看”的前置判断。用户进入 VidMark 的视频默认已经重要，产品目标是降低从“看到视频”到“真正看懂并留下知识资产”的摩擦。

一句话描述：

> VidMark：把必看视频变成可跳转、可翻译、可笔记的中文精读材料。

## 用户场景

用户常在 X 或 YouTube 上关注高质量干货博主和推主。很多视频来自可信推荐源，用户已经决定要看，但真正看懂这些英文长视频通常需要经历复制链接、找字幕、翻译、定位重点、做笔记、保存到 Obsidian 等多个步骤。

VidMark 要把这个流程压缩为一个入口：

1. 用户在浏览器里看到视频。
2. 点击插件按钮或快捷键进入 VidMark 模式。
3. VidMark 自动准备字幕、中文精译、高能片段和笔记工作台。
4. 用户边看边保存关键句、片段和想法。
5. VidMark 把视频卡片写入 Obsidian，供后续检索、复盘和关联旧笔记。

## 设计原则

- 浏览器内完成：在哪里看到视频，就在哪里进入精读。
- 默认视频重要：不把首屏浪费在筛选和推荐上。
- 中文翻译是主体验：翻译要自然、准确、有语境，不只是附加字幕。
- 字幕即导航：每一句字幕都应该能带用户跳回视频现场。
- 笔记绑定时间戳：所有摘录、想法、解释都能回到原片段。
- Obsidian 只负责沉淀：重交互留在浏览器，长期知识资产写入本地 vault。
- 失败路径明确：字幕、翻译、转写、保存任一环节失败时，要告诉用户下一步能做什么。

## MVP 范围

MVP 只覆盖桌面 Chrome/Arc 和本地 Bridge。第一阶段优先打通 YouTube，X 视频作为同一架构下的后续能力。

MVP 包含：

- YouTube 视频识别和 VidMark 入口。
- 视频、字幕句、翻译句、高能片段、视频笔记的数据模型。
- YouTube 字幕读取和失败状态。
- 通过本地 Bridge 调用 Codex 生成中文精译。
- VidMark 阅读器界面：视频、双语字幕、点击跳转、当前句高亮。
- 高能片段列表：观点、案例、方法论、金句、争议点。
- Obsidian 视频卡片写入。
- 使用说明和更新日志。

MVP 暂不包含：

- 移动端。
- 账号系统。
- 云端同步。
- 社交分享。
- 完整的跨平台视频下载器。
- 对所有 X 视频的稳定转写保证。
- 自动批量处理播放列表。

## 模块拆分

### 1. 视频捕获模块

职责：

- 判断当前页面是否存在可处理视频。
- 对 YouTube 提取视频 ID、URL、标题、频道、播放状态和当前时间。
- 对 X 页面先提取推文 URL、页面标题、可见视频元素和推荐上下文。
- 向 VidMark 阅读器提供统一的视频元数据。

边界：

- 不负责翻译。
- 不负责写 Obsidian。
- 不在 MVP 中承诺所有嵌入视频都能处理。

### 2. 字幕获取模块

职责：

- 优先获取页面上已有的 YouTube 字幕轨道。
- 把字幕归一化为带时间戳的句子列表。
- 记录字幕语言、来源、是否自动字幕、是否缺失。
- 在无字幕时返回可解释的失败状态，为后续 Whisper 转写留接口。

边界：

- MVP 只要求 YouTube 字幕链路可靠。
- X 视频转写在后续阶段通过本地 Bridge 加入。

### 3. 中文精译模块

职责：

- 把字幕按连续时间块切分。
- 调用本地 Bridge，由 Codex 生成自然中文精译。
- 保留原句、译文、时间戳和块级上下文。
- 维持术语一致性，避免逐句割裂翻译。

翻译要求：

- 中文优先自然表达。
- 专有名词首次出现保留英文。
- 对口语、比喻、梗和语气做语境化处理。
- 不添加原视频没有的观点。

### 4. 高能片段模块

职责：

- 基于原字幕和中文精译识别重点片段。
- 输出片段标题、类型、摘要、起止时间、证据句。
- 支持点击片段跳转视频。

片段类型：

- 观点
- 案例
- 方法论
- 金句
- 争议点
- 行动建议

### 5. 视频阅读器 UI 模块

职责：

- 在浏览器内提供 VidMark 阅读器。
- 左侧显示视频，右侧显示双语字幕。
- 支持字幕随播放滚动、当前句高亮、点击字幕跳转。
- 支持在“字幕”“高能”“笔记”三个视图之间切换。

交互约束：

- 默认不跳出当前浏览器工作流。
- 控件密度要适合长时间学习，不做营销页式布局。
- 出错时在当前面板内显示可操作状态。

### 6. 笔记模块

职责：

- 保存当前字幕句、片段和用户想法。
- 每条笔记绑定视频 URL、时间戳、原文、译文。
- 支持把 AI 解释或用户追问结果附加到笔记。

边界：

- MVP 先做本地内存态和单次保存。
- 后续再做跨页面恢复和草稿缓存。

### 7. Obsidian 写入模块

职责：

- 生成标准视频卡片 Markdown。
- 写入 Think Anytime vault 的 VidMark 目录。
- 包含来源、视频信息、高能片段、双语摘录、用户笔记、后续问题。
- 更新 VidMark 索引和更新日志。

建议 vault 结构：

```text
50-VIDMARK/
├── inbox/
├── videos/
├── clips/
└── index.md
```

### 8. 本地 Bridge 模块

职责：

- 提供 VidMark 专用 API。
- 调用 Codex 进行字幕精译、高能片段提取和卡片生成。
- 后续接入 `yt-dlp` 与 Whisper。
- 复用 Think Anytime 的鉴权、配置、vault 写入和 trace 能力。

## 数据流

```text
浏览器当前页
  -> 视频捕获模块
  -> 字幕获取模块
  -> 本地 Bridge
  -> 中文精译模块
  -> 高能片段模块
  -> VidMark 阅读器
  -> 用户笔记
  -> Obsidian 视频卡片
```

## 测试策略

- 数据模型和纯函数使用 Node 单元测试。
- Bridge API 使用请求级测试，覆盖成功、失败和非法输入。
- 浏览器入口使用最小可复现 DOM 测试，优先测试页面识别和状态转换。
- UI 在本地浏览器中手动验证 YouTube 视频页，检查字幕展示、点击跳转和保存链路。
- 每个阶段提交前运行与改动范围匹配的检查命令；MVP 收尾阶段运行完整检查。

## 风险与应对

- YouTube 字幕接口变动：把字幕获取封装为独立模块，失败时不影响阅读器其他状态。
- X 视频抓取不稳定：MVP 不把 X 转写作为首个可用闭环，先完成识别和回退状态。
- 中文翻译质量波动：使用块级上下文、术语表和结构化输出校验。
- 长视频成本过高：按需翻译当前窗口和高能片段，后续再做全量后台处理。
- Obsidian 写入污染：VidMark 使用独立目录和明确 frontmatter，不混入普通卡片。

## 成功标准

MVP 完成时，用户可以在 YouTube 视频页一键进入 VidMark，看到双语字幕，点击字幕跳转视频，查看 AI 标出的高能片段，保存至少一条带时间戳的笔记，并在 Obsidian 中得到一张结构清晰的视频卡片。
